# Inbox Context Integration - 设计方案

## 背景

Inbox 功能的定位转变：从"待办清单"到"长期参考库/碎片库"。

核心场景：
- 用户随时记录想法、待办、碎片
- AI 在其他对话中识别相关性并建议融入
- 融入后 inbox 项被删除

---

## 方案 A：AI 主动扫描（已实现）

**实现方式**：
1. Inbox 内容同步到 `INBOX.md` 文件
2. 在 AGENTS.md 中引用，AI 每次 session 自动加载
3. AI 判断相关性，主动建议"要不要一起讨论？"
4. 用户确认融入后，调用 API 删除 inbox 项

**优点**：
- 自动化，无需用户手动操作
- AI 有完整的上下文可以判断相关性

**缺点**：
- 每次 session 都要读一遍，可能干扰
- 浪费 token（如果 inbox 内容很多）

**当前状态**：已实现基础功能，待测试

---

## 方案 B：分类标记

**实现方式**：
1. Inbox item 增加字段区分类型：`todo` / `note`
2. `todo` 类：AI 主动提醒，有相关性就建议融入
3. `note` 类：只在相关时作为背景知识引用，不主动问

**优点**：
- 减少干扰，用户可以选择哪些需要 AI 关注
- `note` 类可以长期保留作为参考

**缺点**：
- 用户需要手动分类，增加操作负担
- 可能记的时候不想分，后来忘记分类

**未来方向**：
- 可以和方案 A 结合，用 AI 自动判断类型
- 或者用自然语言关键词自动分类（"提醒我"、"之后要做" → todo）

---

## 方案 C：自然语言判断

**实现方式**：
1. 不强制分类，AI 自己判断 inbox 项的性质
2. 如果是"提醒我"、"之后要做"这类 → 主动问要不要融入
3. 如果是纯想法、思考记录 → 作为背景 context，不强推

**优点**：
- 用户无需手动操作
- 更智能，AI 理解用户意图

**缺点**：
- AI 判断可能不准确
- 实现复杂度较高

**未来方向**：
- 可以在 inbox 存储时增加隐式标记（AI 分析后添加）
- 或者对话时根据上下文动态判断
